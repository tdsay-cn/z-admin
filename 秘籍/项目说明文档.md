# Z-Admin 项目说明文档

> 本文档为项目开发规范和详细技术说明。项目简介、快速上手、环境搭建请参考根目录下的 `README.md`。
>
> 本后端项目需配合前端项目 **vue-z-admin** 使用。**vue-z-admin** 是一个开箱即用的中后台前端解决方案，基于 `vue-element-admin` 模板构建，内置登录鉴权、动态路由、权限控制等功能。

## 1. RESTful 应用说明

### 1.1 项目架构

本项目采用标准的 Spring Boot RESTful 架构，遵循分层设计原则：

```
controller (控制层) → service (业务层) → dao (数据访问层) → database (数据库)
```

### 1.2 Controller 层规范

Controller 层负责处理 HTTP 请求和响应，遵循 RESTful 风格：

| 操作 | 注解 | 示例路径 |
|------|------|----------|
| 新增 | `@PostMapping` | `/admin/system/user` |
| 查询列表 | `@GetMapping("/list")` | `/admin/system/user/list` |
| 查询单个 | `@GetMapping("/{id}")` | `/admin/system/user/1` |
| 更新 | `@PutMapping("/{id}")` | `/admin/system/user/1` |
| 删除 | `@DeleteMapping("/{id}")` | `/admin/system/user/1` |

### 1.3 请求参数规范

- **路径参数**：使用 `@PathVariable` 注解
- **查询参数**：使用 `@RequestParam` 注解或直接绑定到对象
- **请求体参数**：使用 `@RequestBody` 注解，配合 `@Valid` 进行参数校验

### 1.4 参数校验方式

项目基于 `jakarta.validation` (Spring Boot 3.x) 实现参数校验，支持**内置注解**和**自定义注解**两种方式。

#### 1.4.1 常用内置校验注解

| 注解 | 说明 | 示例 |
|------|------|------|
| `@NotNull` | 不能为 null | `@NotNull(message = "ID不能为空")` |
| `@NotBlank` | 不能为 null 且必须包含非空白字符 | `@NotBlank(message = "用户名不能为空")` |
| `@NotEmpty` | 不能为 null 且长度 > 0 | `@NotEmpty(message = "列表不能为空")` |
| `@Size(min, max)` | 字符串/集合长度范围 | `@Size(min = 6, max = 20, message = "密码长度6-20位")` |
| `@Min(value)` | 数值最小值 | `@Min(value = 1, message = "ID必须大于0")` |
| `@Max(value)` | 数值最大值 | `@Max(value = 100, message = "不能超过100")` |
| `@Email` | 邮箱格式校验 | `@Email(message = "邮箱格式不正确")` |
| `@Pattern(regexp)` | 正则表达式匹配 | `@Pattern(regexp = "^1[3-9]\\d{9}$", message = "手机号格式不正确")` |

#### 1.4.2 自定义校验注解

项目提供自定义校验注解，通过 `@Constraint` 关联校验器实现。

**定义自定义注解（以 `@Phone` 为例）：**

```java
@Documented
@Constraint(validatedBy = PhoneValidator.class)
@Target({ElementType.FIELD, ElementType.PARAMETER})
@Retention(RetentionPolicy.RUNTIME)
public @interface Phone {
    String message() default "手机号格式不正确";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}
```

**实现校验器（以 `PhoneValidator` 为例）：**

```java
public class PhoneValidator implements ConstraintValidator<Phone, String> {

    private static final String PHONE_REGEX = "^1[3-9]\\d{9}$";

    @Override
    public boolean isValid(String phone, ConstraintValidatorContext context) {
        if (DataUtils.isEmpty(phone)) {
            return true;
        }
        return phone.matches(PHONE_REGEX);
    }
}
```

> 自定义校验器中，空值一般返回 `true`（放行），由 `@NotBlank` 等注解负责必填校验，做到单一职责。

#### 1.4.3 项目已提供的自定义注解

| 注解 | 校验器 | 说明 |
|------|--------|------|
| `@Phone` | `PhoneValidator` | 手机号格式校验（1开头，11位数字） |
| `@ValidId` | `ValidIdValidator` | ID校验（不能为空且必须大于0） |

#### 1.4.4 使用示例

**在 Form/DTO 类中使用：**

```java
public class UserForm {

    @NotBlank(message = "用户名不能为空")
    @Size(min = 3, max = 20, message = "用户名长度3-20位")
    private String username;

    @Phone(message = "请输入有效的手机号")
    private String phone;

    @ValidId(message = "角色ID无效")
    private Long roleId;
}
```

**在 Controller 中使用：**

```java
@PostMapping
public Result<Void> add(@RequestBody @Valid UserForm form) {
    // 参数校验失败时会抛出 MethodArgumentNotValidException，
    // 由全局异常处理器统一捕获并返回友好提示
}
```

#### 1.4.5 校验异常处理

参数校验失败会抛出 `MethodArgumentNotValidException`，项目通过全局异常处理器统一处理：

1. 从异常中提取所有校验错误信息
2. 返回 `Result.fail(...)` 格式的错误响应
3. 前端可直接展示 `msg` 字段中的提示内容

## 2. 各种 O 之间转换、使用工具 Util 说明

### 2.1 对象转换工具

项目使用两种方式进行对象转换：
- **BeanUtils**：运行时反射拷贝，简单灵活但性能一般，适用于简单场景
- **MapStruct**：编译期生成转换代码，类型安全且高性能，适用于频繁转换的场景

### 2.2 BeanUtils 使用

```java
BeanUtils.copyProperties(source, target);
BeanUtils.copyProperties(source, target, "id", "createTime");
```

### 2.3 MapStruct 使用

**基本用法（参考 `PermissionConvert`）：**

```java
@Mapper(componentModel = "spring", unmappedTargetPolicy = ReportingPolicy.IGNORE)
public interface PermissionConvert {
    PermissionTreeNodeVo toTreeNodeVo(PermissionTreeDto dto);
    List<PermissionTreeNodeVo> toTreeNodeVoList(List<PermissionTreeDto> dtoList);
}
```

**核心配置：**

| 配置 | 说明 |
|------|------|
| `componentModel = "spring"` | 生成 Spring Bean，可 `@Autowired` 注入 |
| `unmappedTargetPolicy = IGNORE` | 未匹配字段不报错 |

**常用映射场景：**

| 场景 | 用法 |
|------|------|
| 字段名相同 | 自动匹配，无需配置 |
| 字段名不同 | `@Mapping(source = "userName", target = "username")` |
| 忽略字段 | `@Mapping(target = "password", ignore = true)` |
| 日期格式化 | `@Mapping(target = "createTime", dateFormat = "yyyy-MM-dd HH:mm:ss")` |
| 嵌套字段 | `@Mapping(source = "user.name", target = "userName")` |
| 多源对象 | `toInfoVo(SystemUser user, SystemRole role)` 合并映射 |

**使用方式：**

```java
@Autowired
private PermissionConvert permissionConvert;

List<PermissionTreeNodeVo> voList = permissionConvert.toTreeNodeVoList(dtoList);
```

**文件规范：** 所有转换接口统一放在 `entity.convert` 包下，命名为 `{业务名}Convert`。

### 2.4 对象类型说明

| 类型 | 说明 |
|------|------|
| PO | 持久化对象，对应数据库表 |
| VO | 视图对象，返回给前端 |
| DTO | 数据传输对象，服务间传输 |
| Form | 表单对象，接收前端参数 |
| Param | 查询参数对象 |

## 3. 使用 Page 自己实现 Count

### 3.1 分页参数 BasePageParam

项目使用自定义分页参数类，关键点：
- `new Page<>(current, size, false)` - 禁用自动 count 查询
- `setOptimizeCountSql(false)` - 优化 count SQL

### 3.2 分页查询流程

1. 获取分页对象（不执行 count）
2. 执行分页查询（只查询当前页数据）
3. 手动执行 count 查询（如果需要总数）
4. 转换为 VO

### 3.3 性能优化说明

- **禁用自动 count**：避免 MyBatis Plus 自动生成 count SQL
- **按需 count**：根据实际需求决定是否执行 count 查询

### 3.4 分页返回结果

| 字段 | 说明 |
|------|------|
| records | 当前页数据列表 |
| total | 总记录数 |
| size | 每页显示数量 |
| current | 当前页码 |
| pages | 总页数 |

## 4. 多数据源

### 4.1 数据源配置

项目使用 `dynamic-datasource` 实现多数据源支持，配置文件：`application-dev.yml`

| 数据源 | 配置名 | 用途 |
|--------|--------|------|
| 主库 | master | 写操作 |
| 从库 | slave | 读操作 |

### 4.2 使用默认数据源

默认使用 `master` 数据源，无需额外配置。

### 4.3 指定数据源

使用 `@DS` 注解指定数据源：

```java
@DS("master")   // 使用主库
@DS("slave")    // 使用从库
```

### 4.4 事务管理

- 单数据源事务：配合 `@Transactional` 和 `@DS` 使用
- 多数据源事务：需要使用分布式事务（本项目暂未实现）

### 4.5 读写分离建议

| 操作类型 | 数据源 |
|----------|--------|
| 写操作（insert/update/delete） | master |
| 读操作（select） | slave |

## 5. 登录和鉴权（详细说明）

> 关于权限模型的概述请参考 `README.md`。本章为详细的技术实现说明。

### 5.1 JWT 工具类

项目使用 `JwtUtil` 工具类进行 Token 的生成和解析：

| 方法 | 说明 |
|------|------|
| `JwtUtil.generate(username)` | 生成 JWT Token，过期时间 1 天 |
| `JwtUtil.parse(token)` | 解析 JWT Token，失败返回 null |

### 5.2 Spring Security 配置

主要配置项：

| 配置项 | 说明 |
|--------|------|
| `csrf.disable()` | 关闭 CSRF |
| `sessionCreationPolicy(STATELESS)` | 无状态模式 |
| `cors()` | 开启跨域 |
| `addFilterBefore(authFilter, ...)` | 添加认证过滤器 |
| `addFilterBefore(MyAnonymousAuthenticationFilter, ...)` | 添加匿名过滤器 |

### 5.3 认证过滤器 AuthFilter

`AuthFilter` 用于拦截请求并验证 Token：
1. 从请求头获取 `Authorization` Token
2. 使用 `JwtUtil.parse()` 解析 Token
3. 从 Redis 缓存查询用户信息
4. 验证 Token 有效性，将用户信息放入 `SecurityContext`

### 5.4 匿名鉴权过滤器 MyAnonymousAuthenticationFilter

`MyAnonymousAuthenticationFilter` 用于处理未登录用户的匿名访问权限：
1. 从数据库查询匿名访问权限列表
2. 创建包含匿名权限的认证对象

### 5.5 请求鉴权流转过程

#### 5.5.1 鉴权流程总览

```
请求进入 → CorsFilter → MyAnonymousAuthenticationFilter → AuthFilter → MyAuthorizationManager → 目标 Controller
```

#### 5.5.2 详细流转说明

| 步骤 | 类名 | 作用 |
|------|------|------|
| 1 | CorsFilter | 处理跨域请求 |
| 2 | MyAnonymousAuthenticationFilter | 设置匿名用户权限 |
| 3 | AuthFilter | 验证 JWT Token |
| 4 | MyAuthorizationManager | 判断用户是否有权限访问 |
| 5 | MyEntryPoint | 认证失败时返回错误 |

#### 5.5.3 登录用户请求流程

1. 请求进入，经过 CorsFilter
2. MyAnonymousAuthenticationFilter 创建匿名认证对象
3. AuthFilter 从请求头获取 Token，解析验证
4. Token 有效：从 Redis 获取用户信息，更新 SecurityContext
5. Token 无效：保留匿名认证对象
6. MyAuthorizationManager 检查用户权限
7. 有权限：访问目标 Controller；无权限：MyEntryPoint 返回 40101

#### 5.5.4 匿名用户请求流程

1. 请求进入，经过 CorsFilter
2. MyAnonymousAuthenticationFilter 创建包含匿名权限的认证对象
3. AuthFilter 发现 Authorization 为空，跳过验证
4. MyAuthorizationManager 检查匿名权限
5. 有权限：访问目标 Controller；无权限：MyEntryPoint 返回 40101

#### 5.5.5 MyAuthorizationManager 权限检查逻辑

1. **超级管理员判断**：权限包含 `-1` 的用户直接放行
2. **查找权限配置**：从 Redis 获取所有权限配置，查找请求路径对应的权限
3. **登录级别检查**：如果是 LOGIN 级别且用户已登录，直接放行
4. **权限匹配**：检查用户是否拥有该权限

#### 5.5.6 权限级别说明

| 权限级别 | 说明 |
|----------|------|
| ANONYMOUS | 匿名可访问 |
| LOGIN | 登录即可访问 |
| AUTHORITY | 需要具体权限 |

#### 5.5.7 过滤器注册顺序

- `MyAnonymousAuthenticationFilter` 必须在 `AuthFilter` 之前
- `AuthFilter` 必须在 `UsernamePasswordAuthenticationFilter` 之前

**顺序说明：**
1. `MyAnonymousAuthenticationFilter` 必须先执行，确保所有请求都有一个初始的认证对象
2. `AuthFilter` 在其后执行，用登录用户信息覆盖匿名认证对象
3. `MyAuthorizationManager` 在过滤器之后执行，进行最终权限校验

### 5.6 获取登录用户信息

使用 `LoginUtil.getLoginUser()` 获取当前登录用户信息，从 `SecurityContext` 中获取 `UserDetail` 对象。

### 5.7 登录方式说明

#### 5.7.1 为什么不使用 Spring Security 默认登录方式

项目采用自定义登录接口方式，而非 Spring Security 默认方式（继承 `UsernamePasswordAuthenticationFilter`），原因：

| 对比项 | Spring Security 默认方式 | 自定义登录接口方式 |
|--------|------------------------|-------------------|
| 灵活性 | 受框架约束 | 完全自主控制 |
| 返回格式 | 固定格式 | 统一使用 `Result` 格式 |
| 登录逻辑 | 需要实现 `UserDetailsService` | 自定义业务逻辑 |
| 扩展性 | 扩展需要继承多个类 | 可自由添加额外逻辑 |

#### 5.7.2 LoginFilter（未使用）

项目中有一个被注释掉的 `LoginFilter`，是 Spring Security 默认登录方式的示例。

**Spring Security 默认登录方式的使用步骤：**
1. 实现 `UserDetailsService` 接口
2. 配置登录 URL
3. 使用 `AuthenticationManager.authenticate()` 进行认证
4. 在 `successfulAuthentication` 中处理登录成功逻辑

#### 5.7.3 自定义登录接口（推荐方式）

通过 `AuthController` 提供登录服务：

| 接口 | 方法 | 说明 |
|------|------|------|
| `/admin/auth/login` | POST | 用户登录 |
| `/admin/auth/info` | GET | 获取用户信息 |
| `/admin/auth/logout` | POST | 用户退出 |

#### 5.7.4 自定义登录流程

1. 参数校验（@Valid）
2. 查询用户信息
3. 验证密码
4. 检查用户状态
5. 生成 JWT Token
6. 缓存用户信息到 Redis

### 5.8 权限管理（详细实现）

#### 5.8.1 权限分配方式

系统支持两种权限分配方式，可独立使用也可混合使用：

| 分配方式 | 说明 | 适用场景 |
|---------|------|---------|
| **通过角色分配** | 用户 → 角色 → 权限集合 | 批量用户管理、标准化岗位权限 |
| **直接分配权限** | 用户 → 权限 | 个性化权限、临时授权 |

最终用户权限 = 用户直接分配的权限 ∪ 所属角色包含的权限

#### 5.8.2 权限检查流程

权限配置通过 `MyAuthorizationManager` 实现自定义鉴权逻辑，所有请求都通过该管理器进行权限校验。

**权限校验的关键步骤：**
1. 获取当前登录用户
2. 加载用户**直接配置的权限**列表
3. 加载用户**所属角色映射的权限**列表
4. 将两部分权限合并去重，得到用户最终权限集合
5. 判断用户权限集合中是否包含请求路径所需的权限标识

### 5.9 认证失败处理

项目使用 `MyEntryPoint` 处理认证失败情况，返回统一的 `Result` 格式错误信息。

### 5.10 Token 过期处理

当 Token 过期或无效时，系统返回错误码 `40101`（认证失败）。
